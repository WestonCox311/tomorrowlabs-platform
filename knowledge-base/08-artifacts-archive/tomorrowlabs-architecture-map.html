Architectural Principles

Six commitments encoded structurally.

Principles that hold across the entire architecture, not specific to any layer. These are the architectural expressions of the data philosophy commitments.

01

Every fact carries provenance.

The sources table is referenced by every observation, claim, and assessment. No floating facts. Citation is structural, not aspirational.

02

Every fact carries confidence.

The confidence_level enum is on every table making claims. High, medium, low, estimated. Hidden gaps become visible. Estimated data is honest data.

03

Consent sits above operational logic.

Layer 3 reads from consent_records but cannot override it. Operational pressure cannot erode community commitments because the schema prevents it.

04

Time-series never overwrite.

Observational tables append, never update. A row added in 2026 stays unchanged in 2030. Trajectory becomes visible because history is preserved.

05

Community positions are first-class.

When community classifications differ from academic ones, the schema records both. Disagreement is held structurally, not flattened.

06

Decisions close the loop.

Every significant decision in decision_log is followed by an outcome assessment in decision_outcomes. The organization remembers its reasoning and tests it against reality.

Build History

How this got built, migration by migration.

The architecture was built in deliberate phases over six SQL migrations. Each migration produces value while maintaining architectural coherence with what came before.

000
Initial language database

21 tables. Glottolog-anchored language spine with depth across demand, tech, culture, partnership.

001
Omnilingual & CV updates

Schema extensions for Meta Omnilingual ASR support and Mozilla Common Voice tracking.

002
Language depth

Vitality assessments, transmission tracking, orthographies, documentation status, revitalization programs.

003
Phase 1 Spine extension

Places, organizations, and communities as canonical reference entities. The architecture takes shape.

004
Layer 2 Observational

Time-series tracking across migration, sector events, tech evolution, funding flows, regulatory changes.

005
Layer 3 Operational

Programs, deployments, products, content, consent infrastructure, finance, team, risk registry.

006
Layer 4 Decision Support

Decision protocols, outcomes tracking, AI synthesis audit, strategic views, dashboard inventory.

Ongoing

Data flows up the stack. Decisions flow down.
How information moves through the architecture in practice.
i.
Reference is curated.

Glottolog, GeoNames, and external sources feed Layer 1 through deliberate research. Slow, careful, citation-grade.

ii.
Observations accumulate.

Layer 2 grows as the world is observed. Each row is a moment in time, never overwritten. Trajectory becomes legible.

iii.
Operations run.

Layer 3 captures active work via integrations with Asana, Stripe, product analytics. References upward but never overrides.

iv.
Decisions emerge.

Layer 4 synthesizes everything below into views and protocols. Decisions made here feed back as outcomes — and the loop closes.

This is the complete architecture as built.

What you see in this document is what TomorrowLabs's data infrastructure actually is — not a prospectus, not a wishlist. Sixty-five tables across four layers, with seven cross-cutting concerns, six SQL migrations of work, and twenty-plus strategic views. The schema is ready. The architecture is real.

What this map doesn't show is the work ahead. Most tables are sparsely populated. Most strategic views are structurally correct but data-poor. The architecture's value compounds with use — each populated row, each executed protocol, each closed decision loop makes the whole more useful.

The next year is about populating what's been designed. Year two is about refining what's been used. The multi-generational horizon is about the architecture outlasting the organization that built it. The structure is designed for that scale of time. The data work begins now.
