Data Architecture v1

How TomorrowLabs organizes what it knows.

Four layers, governed by shared infrastructure, designed to support strategic, operational, and real-time decisions without letting any one of them colonize the others. This is the architectural reference for everything we build.

The Argument

An intelligent organization needs three kinds of data, and three kinds of decisions.

Most organizations conflate their data infrastructure. They build one system that tries to be a research database, an operational tool, and a real-time product backend simultaneously. The result is a system that's mediocre at all three.

TomorrowLabs operates at three timescales that demand fundamentally different data approaches. Strategic decisions — where to expand, which languages to serve — happen annually and require careful, cited, research-grade data. Operational decisions — when to ship, how to price — happen weekly and require accurate current-state data. Real-time decisions — what to show this user right now — happen thousands of times a day and require fast, contextual data.

The architecture below treats these as distinct but connected layers. Each layer has its own characteristics, its own update cadence, its own access patterns. Cross-cutting infrastructure ensures they cohere into a single intelligent system rather than fragment into disconnected silos.

Decision Support

Not separate tables — curated views, dashboards, and AI-assisted analyses that combine data from layers below to answer strategic, operational, and exploratory questions. Question-driven. Audience-specific.

Operational Data

The work TomorrowLabs is currently doing.

Active partnerships, deployments in flight, products in market, content libraries, financial transactions, user activity. High-velocity, organization-private, integrated with operational tools. This is the layer that makes the company run.
cadence · daily to hourly · sources · Asana, Stripe, product DBs, internal tools · access · controlled

02
Observational Data

What is actually happening in the world.

Time-series data about populations, languages, markets, infrastructure, partnerships. Never overwritten — always appended. Each row is a snapshot at a point in time. This is how trajectory becomes legible and how impact becomes measurable.
cadence · weekly to yearly · sources · censuses, surveys, partner reports, sensor data · format · longitudinal
Serves Trajectory analysis, impact measurement, trend detection across years.
01
Reference Data

The slow-moving truth about the world.

Canonical entities: languages, places, organizations, frameworks, standards. Slow-moving. Curated carefully. Cross-referenced to authoritative external sources. The substrate every other layer references. Multi-generationally durable.

Each layer references the layers below it, never above. Layer 1 is foundation; Layer 4 sits at the surface where decisions get made. The cross-cutting strip is not a layer in the same sense — it's the connective tissue.

The Inventory

What actually lives in each layer.

A working sketch of the entities and tables per layer. Final counts will evolve as the model gets built, but the order of magnitude is roughly 50–70 tables across the whole system. The language database already built (21 tables) sits inside Layer 1 and Layer 2.

Layer 01 · Reference

The canonical entities.

  • Languages already built — Glottolog-anchored, hierarchical, granularity-aware.
  • Places phase 1 — GeoNames + indigenous territory overlays, hierarchical from country to neighborhood.
  • Organizations phase 1 — Partners, funders, peer orgs, governments. The ecosystem we work in.
  • Communities — Diaspora networks, religious communities, professional networks. Cross-cuts languages and places.
  • People-roles — Community leaders, researchers, partner contacts. Tracked at the level of role, not personal identity.
  • Frameworks & Classifications — EGIDS, vitality scales, regulatory categories, standards we reference.
  • Standards & Protocols — Data formats, identifier systems, methodology references.
Layer 02 · Observational

The world over time.

  • Demographics — Populations, age structures, migration patterns. Per place, per community, per language.
  • Economic indicators — Purchasing power, currency, market signals, price benchmarks.
  • Infrastructure — Digital divide metrics, electrification, postal reliability, logistics quality.
  • Educational outcomes — Literacy, enrollment, language-of-instruction status.
  • Vitality & transmission partially built — Already designed for languages; same pattern for places & communities.
  • Cultural calendar — Holidays, seasons, school terms, harvest cycles.
  • Regulatory environment — Data protection laws, content regulations, indigenous data sovereignty.
  • Health & safety — Disease prevalence, conflict zones, child protection infrastructure.
  • Partnership health — Trust metrics, relationship state changes over time.
Layer 03 · Operational

Active work.

  • Active partnerships — Field partners, MOU status, deployment readiness.
  • Programs & deployments — What's running where, with whom, against what milestones.
  • Products in market — Babagigi, LDL, etc. State, performance, user counts.
  • Content libraries — Storybooks, audio, curriculum, visual assets. With rights and licensing.
  • Users & consent — Account data, consent state, communication preferences.
  • Financial operations — Revenue, expenses, grants in flight, benefit-sharing flows.
  • Team & capacity — Who's doing what, what's blocked, what's at risk.
  • Communications archive — With partners, communities, funders.
  • Technical infrastructure — Deployed models, vendor contracts, integration status.
Layer 04 · Decision Support

Synthesized views.

  • Strategic dashboards — Board-level, quarterly. Portfolio health, mission progress.
  • Operational dashboards — Team-level, weekly. What needs attention, what's drifting.
  • Decision protocols — Formalized recurring questions. "Should we expand to X?" with reproducible methodology.
  • AI-assisted analyses — LLM-mediated synthesis across layers, for exploratory questions.
  • Public-facing reports — Transparency mechanisms for funders, communities, the field.
  • Partner-facing views — What each community partner can see about TomorrowLabs's work in their context.

What the architecture alone can't do.

Three pieces of infrastructure that sit outside the data model itself but determine whether the model becomes intelligence or noise.

Data governance function

Someone whose job is data quality, identifier conflicts, consent management, access audits. Initially Weston; eventually a dedicated role. Without this, the system degrades within two years.

Decision-making protocol

When do we actually consult the data? How does it flow into deliberation? Organizations that build great data infrastructure and then make decisions by gut feel haven't become data-intelligent — they have decoration.

Community accountability

How do partner communities review what we hold about them? How do they correct errors or revoke consent? This is the governance the pressure-test critique demanded. It cannot be retrofitted.

The Tension We Must Architect Around

When humanitarian and commercial logics conflict, which layer wins?

Layer 3 (operational) is where commercial pressure lives. Subscription metrics, conversion goals, retention math. Layer 1 and 2 (reference and observational) are where humanitarian commitments live. Consent state, transmission risk, community position.

In practice these pull in different directions. When Layer 3 says "send this push notification to drive conversion" and Layer 1/2 says "this community has opted out of marketing communications," only one can prevail.

The architectural answer is encoded in the structure itself, not in policy alone. The cross-cutting consent and rights layer sits above operational logic and cannot be overridden by it. Layer 3 reads Layer 1/2; Layer 1/2 never reads from Layer 3. The schema enforces what the values claim. If the architecture doesn't enforce it, operational pressure will erode it over time.

The Phasing

How we build this without trying to build it all at once.

The temptation is to design everything before populating anything. Resist this. The smart approach is deliberate phases that produce value at each step while keeping the architectural model coherent.

Phase 01

The Spine

Q3 2026 · 3 months

Extend Layer 1: add places and organizations alongside languages, with the same care. These three reference entities unlock cross-referenceable infrastructure for almost everything else.

Phase 02

Observational Depth

Q4 2026 – Q1 2027 · 6 months

Build Layer 2 tables for the dimensions most critical to current decisions: demographics, market readiness, digital infrastructure. Populate sparsely with high-confidence data first.

Phase 03

Operationalize

Q2 2027 onwards

Connect Layer 3 to actual operational systems via API integration: Asana, Stripe, product analytics, Common Voice contributions. Mostly integration work, faster than it sounds.

Phase 04

Decision Support

Q3 2027 · ongoing

Build Layer 4 views and dashboards that answer specific strategic questions. This phase never ends — it adapts continuously as the organization learns which questions matter most.

Why this shape

The architecture is the philosophy made operational.

The four-layer model is not arbitrary. It is the structural expression of what TomorrowLabs believes about data: that reference data deserves slow, careful curation; that observational data must hold trajectory honestly; that operational data must serve the work without colonizing the values above it; and that decision support is the layer where data becomes intelligence.

The cross-cutting infrastructure — sources, consent, audit, identity, access — is where the philosophy stops being a document and becomes enforcement. An organization that says it cares about consent without encoding consent structurally is making a promise it cannot keep. The architecture is the keeping of the promise.

Every subsequent technical decision TomorrowLabs makes will reference this document. New databases attach to a layer. New entities get categorized. New decisions get traced back to which layer they draw from. The architecture is the constitution of the data work.
