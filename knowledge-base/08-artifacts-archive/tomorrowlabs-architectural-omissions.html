TomorrowLabs · Architectural Accountability
Omissions Documented · v1

What we chose not to build, and why.

The architecture is more honest when its deliberate omissions are visible. This document catalogs the tables we did not build at each layer, organized by the reasoning behind each decision.

Why this document exists. Any architecture is defined as much by what it includes as by what it excludes. The data architecture we built has 65+ tables. The architecture we could have built had at least 30 more. Some of those omissions are deliberate design choices we'd defend. Some are scope decisions made to ship. A few are gaps we should acknowledge honestly rather than disguise.

The four reasons something didn't get built. Each omission below is tagged with one of four reasons: deferred (the right table, the wrong time), scope (worth building eventually, not now), avoid-rebuild (something else does this better and we should integrate, not duplicate), or honest-gap (we should have built it and didn't, or the question itself is too hard for the current team to answer alone).

The four reasons
Deferred The right table, the wrong time. Will be built when conditions are right or specific work begins.
Scope Worth building eventually. Not necessary for current decisions or current scale of work.
Avoid Rebuild An existing system does this better. Integration is the right answer, not duplication.
Honest Gap We should have built it, or the question is too hard for the current team alone. Named to make accountable.
01

Layer 01 · Reference

What we didn't build in the spine
people
Honest Gap

A canonical table for individuals — community leaders, researchers, partner contacts, advisors, donors. The most consequential omission in the architecture.

The reason for the gap is technical and political at once. Tracking individuals properly requires GDPR-grade consent infrastructure, privacy controls, deletion mechanisms, and access policies that a four-person startup cannot reasonably implement alongside everything else. Building a half-baked people table would create liability and would breach the consent commitments encoded elsewhere in the architecture.

Resolution path: Build it when the org has capacity for a proper privacy infrastructure design, with legal review. For now, individual references in other tables (relationship contacts, communication participants, team members) hold names as text without aggregation. This is deliberately inefficient — it makes querying across individuals hard, which prevents accidental surveillance use.

frameworks_and_classifications
Deferred

A meta-table holding canonical references to external classification systems (EGIDS scale, UNESCO vitality scale, ELP classifications, ISO standards, regulatory categories). Mentioned in the architecture overview but not built.

Currently these frameworks are referenced via enums and free-form text in the tables that use them. That works for now. When we have more than a handful of frameworks active and need to track which versions we're using, when standards were updated, and which assessments reference which versions, the meta-table becomes useful.

Resolution path: Build when the first framework version conflict surfaces (e.g., EGIDS gets revised and we need to know which assessments used which version).

standards_and_protocols
Deferred

A table for data format standards, identifier systems, file formats, and protocols TomorrowLabs commits to supporting. Mentioned in the architecture overview but not built.

The standards we're committed to (Glottocode, ISO 639-3, GeoNames, IANA timezones, ISO 4217 currency codes, ISO 3166 country codes, Unicode blocks, CLDF for linguistic data) are currently documented in the data philosophy and reference library documents. Centralizing them in a queryable table is useful but not urgent.

Resolution path: Build when we need machine-readable versioning of our standards commitments — probably when first formal data archive partnership is established.

indigenous_territories (as distinct table)
Scope

A dedicated table for indigenous territorial boundaries with full integration to native-land.ca. We chose to handle this within the places table via the granularity = 'indigenous-territory' enum value and the native_land_ca_id field.

A standalone table would let us hold richer data — sovereignty claims, boundary contestation, historical territory shifts, tribal enrollment requirements, treaty references. For TomorrowLabs's current work this is overkill, but for any organization deeply embedded in indigenous-led work, this would need to be a first-class entity.

Resolution path: Build if and when TomorrowLabs becomes deeply embedded in indigenous-led language work where territorial questions are central to operations.

religious_traditions
Deferred

Religious traditions deeply shape how communities transmit language and culture. Buddhist temples in Cambodia. Catholic churches in Guatemala. Islamic schools in diaspora communities. The architecture currently captures religious institutions as organizations but doesn't model traditions themselves as entities.

Resolution path: Build when we need to query "which of our partners work within Theravada Buddhist traditions?" or similar cross-organizational tradition-based queries. Not urgent.

02

Layer 02 · Observational

What we didn't track over time
climate_observations
Honest Gap

Climate change is materially affecting the communities and places TomorrowLabs works in. Sea level rise in Cambodian river delta communities. Drought in Guatemalan highlands. Extreme heat in Mexico City. Wildfire smoke in Portland. The observational layer should track this.

We didn't build it because climate data is voluminous and easily becomes overwhelming. Picking the right subset that's actually decision-relevant — without becoming a climate database masquerading as a language preservation project — requires more thought than I gave it.

Resolution path: Add focused climate observation tables when a deployment decision is actively affected by climate considerations. The place_health_safety table has climate_vulnerability_index and recent_climate_events as starter fields, but they're not yet the dedicated observational tracking the issue deserves.

conflict_and_political_events
Honest Gap

Political instability, conflict escalation, regime changes, coups, civil unrest. These materially affect deployment safety, partner trust, and community wellbeing. They are time-series by nature.

We touched this in regulatory_events and place_health_safety but didn't build a dedicated tracking table. The reason is partly squeamishness — TomorrowLabs's instinct is to focus on positive work, and explicitly cataloging conflict feels uncomfortable. That discomfort is exactly why it should be structured: the architecture can't pretend conflict doesn't shape the work.

Resolution path: Add when (not if) a conflict event affects an active partnership or deployment. The table should exist before we need it urgently.

educational_outcomes_longitudinal
Deferred

A dedicated table for tracking educational outcomes in places we deploy. Literacy rates over time, language-of- instruction policy changes, school enrollment trajectories. Currently aggregated within place_demographics and place_infrastructure.

Resolution path: Split out when LDL deployments produce their own educational outcome data and we need to compare against baseline trajectories.

news_and_media_mentions
Avoid Rebuild

Tracking news mentions of TomorrowLabs, partner organizations, or topics relevant to our work. We didn't build this because Google Alerts, Mention.com, and similar services do it better than a custom table would.

Resolution path: Integrate with whichever media monitoring service we settle on, surface alerts into communications when they affect a partnership, otherwise rely on the external service.

community_internal_dynamics
Honest Gap

Communities are not monolithic. The Cambodian-American community in Long Beach has internal generational divides, political differences, religious factions, class stratification. The community_state table touches "generational tension notes" but doesn't seriously model internal community dynamics.

Modeling these well is genuinely hard — and harder still without community co-authorship of the schema. We deferred it but the deferral is the gap, not the solution. This is the kind of work that should not happen without community involvement at the schema design stage.

Resolution path: Pair with community-co-authorship of the v2 data philosophy. Schema design here is itself a relational act, not just a technical one.

economic_indicators_micro_level
Scope

Micro-economic indicators below the regional level — household spending patterns, local market dynamics, informal economy participation, cash-vs-digital payment ratios at neighborhood scale. The current place_economic_indicators goes down to metro level but not below.

Resolution path: Build when commercial product decisions require sub-regional pricing intelligence — probably when Babagigi expands beyond Wave 1 markets.

03

Layer 03 · Operational

What we didn't build in active work
tasks / kanban / project_management
Avoid Rebuild

Granular task tracking. Sprint boards. Subtasks. Dependencies. We didn't build any of this.

Asana already does this. ClickUp does this. Linear does this. Notion does this. Replacing a purpose-built project management tool with custom SQL tables would be wasted engineering effort that competes for budget with mission work.

The architectural commitment: programs and deployments hold links to external tool URLs (asana_project_url, notion_workspace_url). The strategic data layer doesn't replicate operational tool data — it references it.

payments / transactions / accounting
Avoid Rebuild

Detailed transaction-level financial tracking. Invoice management. Payment processing. Reconciliation. We have financial_flows at the strategic level, but not at the transactional grain.

Stripe does payment processing. QuickBooks does accounting. Bill.com does AP. Each is a specialized tool that's better at its job than custom SQL tables would be. The architecture references these via external_reference and source_system fields.

The architectural commitment: financial_flows rolls up the strategic view. Detail lives in the operational financial tools and is referenced by external ID.

product_analytics / user_behavior_events
Avoid Rebuild

Granular user behavior tracking — page views, feature usage, session length, conversion events. Babagigi will generate millions of these events. We did not build tables for them.

Product analytics tools (Mixpanel, Amplitude, PostHog) are purpose-built for high-cardinality event tracking with proper schema flexibility, replay, cohort analysis, and funnel construction. The strategic data layer should never become a product analytics warehouse.

The architectural commitment: User-level engagement summaries roll into user_households and related summary fields. Granular events stay in product analytics. Cross-system queries use external IDs.

grants_pipeline (vs. grants_active)
Deferred

We built grants_active for grants that have been awarded or are at minimum in formal review. We didn't build a separate pipeline table for very-early-stage funder identification, qualification, and prospecting. Those currently live in the funding_landscape table from migration 002 and in the funder_pipeline view.

When TomorrowLabs's grant-seeking effort scales beyond Weston managing it personally, a dedicated pipeline table with CRM-like fields becomes useful. Right now it would be over-engineering.

Resolution path: Build when grant-seeking becomes a dedicated team function rather than a founder-only activity.

incidents / postmortems
Deferred

A dedicated table for incidents — operational failures, security events, partner disputes, deployment crises — and the postmortems that follow. The operational_risks table tracks what could go wrong; this would track what did.

Resolution path: Build when the first significant incident occurs. The structure should not be designed in advance; let the first real incident shape what the table needs.

community_co_governance_records
Honest Gap

The pressure-test document committed to community co-governance mechanisms by Q1 2027 — community review of major decisions, edit access to documents, governance roles in partnerships. The architecture should structurally support this with a dedicated table tracking community governance moments: when communities reviewed decisions, what they said, what changed as a result.

We didn't build it. The reason is that the governance mechanism itself is still being designed with partner communities, and building a schema before the practice exists would presuppose the answer.

Resolution path: Co-design the schema alongside the actual co-governance practice, with at least one partner community involved in the table design itself. This is structural for the philosophy commitments.

vendor_contracts / legal_agreements
Scope

A dedicated table tracking all formal legal agreements — partnership MOUs, vendor contracts, employment agreements, NDAs, grant contracts, license agreements. Currently scattered across infrastructure_dependencies, benefit_sharing_agreements, grants_active, and elsewhere.

Resolution path: Consolidate when the number of formal agreements exceeds what Weston can mentally inventory. Probably 18-24 months out.

internal_knowledge_base
Scope

A structured table for TomorrowLabs's accumulated operational knowledge — playbooks, FAQs, "how we do X" docs, institutional memory. Currently lives in Notion, Google Docs, and individual heads.

Resolution path: The single-founder dependency risk in operational_risks is partly about this gap. The eventual answer is probably better Notion structure plus referenced URLs in our existing tables, not a dedicated knowledge base table. But the operational risk is real.

04

Layer 04 · Decision Support

What we didn't build in synthesis
scenario_models
Deferred

Tables for explicit scenario modeling — "if Babagigi adds Hmong in Wave 3, what does the financial trajectory look like under three demand assumptions?" These would be parameterized projections, stored over time so the model can be revisited and compared against actuals.

Resolution path: Build when first major strategic decision benefits from explicit scenario comparison with structured parameters. Probably tied to fundraising decisions where investors want to see modeled outcomes.

comparative_benchmarks
Deferred

A dedicated table for tracking how TomorrowLabs compares against peer organizations on relevant metrics — revenue per language served, partnership longevity, benefit-sharing percentages, language preservation outcomes. The metric_definitions table has an external_benchmark field but no time-series structure for tracking benchmark drift.

Resolution path: Build when comparative benchmarking becomes a quarterly governance practice.

community_facing_dashboards (as schema)
Honest Gap

The partner_facing_accountability view exists. The dashboards table includes one partner-community-audience entry. But the architecture doesn't yet structure what a community-facing dashboard should actually contain, or how community members would access it, or how their feedback would flow back.

This is partly an architectural gap and partly a UX/governance question. Community-facing dashboards designed without community input would themselves be a betrayal of the philosophy.

Resolution path: Co-design with partner communities. This is the kind of thing that should be built slowly, with one partner first, learning what they actually want to see.

ml_model_performance_tracking
Scope

For AI-assisted decision support, tracking model performance over time — are our LLM syntheses getting more useful or less? Are certain types of questions consistently producing better outputs? The ai_synthesis_runs table records individual runs with review notes, but doesn't yet aggregate performance.

Resolution path: Build when AI-synthesis use is frequent enough to make trend analysis meaningful. Probably 12-18 months of accumulated runs before patterns become statistically interpretable.

public_reports_archive
Deferred

A canonical record of public reports TomorrowLabs has published — annual reports, impact reports, white papers, public dashboards. Currently these would live in content_items with appropriate flags, but a dedicated table would let us track publication cadence, audience reach, and what we committed to publicly.

Resolution path: Build when public reporting becomes a regular practice — probably tied to first annual report.

00

Cross-cutting · Infrastructure

What we didn't build at the foundation
access_control_policies (as schema)
Honest Gap

The architecture references row-level security and access controls repeatedly. It doesn't structurally encode those policies. Currently access decisions would live in application code or database role configuration.

A proper access control schema would track: who can read what tables, who can modify, what aggregations are permitted, what queries are auditable, what data leaves the system and to whom. Without this, the consent commitments in the architecture are honored by policy and developer discipline, not by structure.

Resolution path: Build before the team grows beyond Weston as the only person with database access. Critical infrastructure for organizational integrity.

data_quality_assertions
Deferred

Structured data quality monitoring — assertions about what should be true in the data (e.g., "every active program has at least one deployment", "every active partnership has a relationship record in the last 90 days"), with automated checking and alerting.

The architecture supports this conceptually — confidence levels, validation flags, freshness views — but doesn't have dedicated infrastructure for ongoing assertion checking. Tools like Great Expectations or dbt tests would be the right answer.

Resolution path: Build when the team grows and the database is being modified by more than one person. Quality drift accelerates with team size.

consent_audit_chain
Honest Gap

The consent_records table holds current consent state. It does not hold an immutable audit chain showing every time consent was checked before an operation occurred. For high-stakes consent (community contributions, audio recordings, children's data), the audit chain is what makes the consent commitment enforceable rather than aspirational.

Resolution path: Add immutable consent check logging when first content distribution or research publication occurs. The chain should be append-only, with cryptographic ordering ideally.

identity_resolution_table
Scope

A dedicated table mapping the same real-world entity across multiple identifier systems. "Long Beach Cambodian community" in MPI data, "Long Beach Cambodians" in Census data, "the Long Beach community" in our own records — same entity, different sources, no current bridge table.

Resolution path: Build when first cross-source comparison of the "same" entity produces confusing results. Probably triggered by a specific analytical need.

The omission that matters most

Community co-authorship of the schema itself.

Every table in this architecture was designed by Weston, working with an AI assistant, with no partner community member in the room when the structural choices were made. The pressure-test document called this out: a philosophy written without community co-authorship has the structural problem it claims to be against.

The omissions tagged "honest-gap" above are mostly attempts to surface where this matters most — community internal dynamics, community-facing dashboards, community co-governance records. These should not have been designed alone. The fact that they were — even with the intent of revising them later — reflects the same structural issue.

The next version of this architecture needs community co-authorship of the schema itself, not just review of what was built. Schema design is a relational act. Doing it well requires being in relationship.

This is named here so the omission is accountable rather than invisible.

The architecture is more honest for naming what it isn't.

Sixty-five tables. Thirty-plus deliberate omissions. Each omission was a choice — to defer, to scope, to integrate rather than rebuild, or to acknowledge a gap we didn't close.

The danger of a sprawling architecture is that everything that gets built feels equally intentional. This document pushes back on that. Some things weren't built because we shouldn't have built them. Some weren't built because the tooling that should have built them isn't ours to construct. Some weren't built because we should have and didn't.

The point of writing this down is not to look more sophisticated about our own gaps. It is to make those gaps maintainable — visible enough that they get addressed when conditions permit, and named clearly enough that we can't pretend they were never there.
